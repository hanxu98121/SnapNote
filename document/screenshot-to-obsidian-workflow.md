# Screenshot To Obsidian 本地网页工作流设计

## 目标

做一个只在本机使用的网页工具，用来把一批截图整理成可复制、可保存的 Obsidian Markdown 笔记。

核心不是传统 OCR，而是把图片和用户说明一起发送给支持多模态的大模型，让模型直接完成识别、理解、清洗、结构化和 Markdown 输出。

## 核心使用方式

1. 用户把待处理截图放入 `input_image/`。
2. 打开本地网页应用。
3. 页面自动读取 `input_image/` 中的图片。
4. 用户可以把图片拖拽成不同分组。
5. 每个分组有一段独立的文字说明。
6. 点击生成后，系统把该分组里的所有图片和该分组说明一起发给多模态大模型。
7. 模型返回 Markdown。
8. Markdown 显示在右侧输出栏。
9. 用户可以复制、修改、重新生成或保存到 `output/`。

## 文件夹结构

```text
screenshot-to-obsidian/
├── input_image/        # 用户放入待处理截图
├── input_text/         # 保存每个分组的用户说明和处理状态
├── output/             # 保存模型生成的 Markdown 文件
├── document/           # 项目文档
├── state.json          # 页面状态，可选
└── app/                # 本地网页应用代码，后续实现
```

## 页面布局

页面采用横向三栏结构，每一行代表一个处理分组。

```text
┌──────────────────┬──────────────────────┬──────────────────────┐
│ 图片分组          │ 用户说明              │ Markdown 输出          │
├──────────────────┼──────────────────────┼──────────────────────┤
│ image 1           │ 这几张图怎么处理       │ 模型生成结果            │
│ image 2           │ [生成] [重试]          │ [复制] [保存]           │
├──────────────────┼──────────────────────┼──────────────────────┤
│ image 3           │ 单独说明              │ 单独输出                │
└──────────────────┴──────────────────────┴──────────────────────┘
```

## 分组逻辑

### 单图分组

默认情况下，每张图片进入页面后都是一个独立分组。

```text
group-001: image-a.png
group-002: image-b.png
group-003: image-c.png
```

### 多图合并分组

用户可以把第二张图拖拽到第一张图所在分组。

合并后：

```text
group-001: image-a.png, image-b.png
group-002: image-c.png
```

这表示 `image-a.png` 和 `image-b.png` 是相关内容，使用同一段说明，一起发给大模型。

### 拆分分组

后续可以支持把某张图片从分组中拖出来，重新变成独立分组。

第一版如果时间有限，可以先支持合并，不急着做复杂拆分。

## 三栏职责

| 区域 | 作用 |
| :--- | :--- |
| 左栏 | 显示图片缩略图、文件名、拖拽排序和分组 |
| 中栏 | 输入这个分组的处理说明 |
| 右栏 | 显示模型生成的 Markdown，可编辑、复制、保存 |

## 用户说明设计

每个分组都有一段单独说明，保存到 `input_text/` 或 `state.json`。

示例：

```text
这几张图是同一篇文章的连续截图。请合并理解，不要重复总结。保留核心观点、关键步骤和可执行建议，输出成适合 Obsidian 的 Markdown。
```

另一个示例：

```text
这是小红书店铺推荐截图。忽略点赞、评论、收藏、分享等 UI 信息，只提取店名、地址、人均、推荐菜和注意事项。
```

## 模型请求结构

每次点击生成时，请求内容由三部分组成。

### 系统规则

```text
你是一位专业的个人知识库管理专家，擅长将截图、文本和用户说明整理成适合 Obsidian 管理的 Markdown 笔记。

要求：
1. 输出必须是 Markdown。
2. 标题从二级标题 ## 或三级标题 ### 开始，禁止使用一级标题 #。
3. 禁止使用 Emoji 和装饰性图标。
4. 去除广告、平台按钮、水印、点赞、评论、分享、加载更多等无关 UI 文案。
5. 保留高价值信息，例如名称、作者、价格、地点、时间、步骤、参数、核心观点。
6. 如果存在多项属性、对比信息或结构化数据，优先使用 Markdown 表格。
7. 不要过度扩写。如果截图信息很少，只做简洁整理。
8. 文末添加“使用建议”，包括推荐文件名、建议内部链接和标签。
9. 只输出最终 Markdown，不解释处理过程。
```

### 用户说明

来自中栏输入框。

### 图片内容

来自当前分组的所有图片。

如果一个分组里有多张图片，则按页面排序依次发送给模型。

## API 调用方向

用户提供的豆包 Ark 示例使用 OpenAI SDK 兼容接口。

当前实现中，API Key、Endpoint ID / model、Base URL 从前端页面输入，然后随生成请求发送给本机后端。后端先调用豆包 Ark；未来可扩展为 OpenRouter 或其他 OpenAI SDK 兼容模型。

后续实现可以沿用这个方向：

```python
import os
from openai import OpenAI

client = OpenAI(
    base_url="https://ark.cn-beijing.volces.com/api/v3",
    api_key="<API_KEY_FROM_FRONTEND>",
)

response = client.chat.completions.create(
    model="<YOUR_ENDPOINT_ID>",
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "image_url",
                    "image_url": {
                        "url": "https://example.com/image.jpeg"
                    },
                },
                {"type": "text", "text": "这是哪里？"},
            ],
        }
    ],
)
```

本地图片需要先转换成模型接口可接受的形式。

可选方式：

| 方式 | 说明 | MVP 建议 |
| :--- | :--- | :--- |
| base64 data URL | 把本地图片转成 `data:image/png;base64,...` | 优先尝试 |
| 本地临时 HTTP URL | 本地服务暴露图片地址 | 备选 |
| 上传到对象存储 | 生成公网 URL | 暂不需要 |

## 状态数据结构

每个分组可以保存成这样的结构：

```json
{
  "id": "group-001",
  "images": [
    "input_image/a.png",
    "input_image/b.png"
  ],
  "instruction": "这两张图是同一篇文章的连续截图，请合并整理。",
  "status": "generated",
  "markdown": "## 文章主题\n\n...",
  "outputFile": "output/2026-05-01-文章主题.md",
  "updatedAt": "2026-05-01T12:00:00"
}
```

状态值建议：

| 状态 | 含义 |
| :--- | :--- |
| `pending` | 尚未生成 |
| `generating` | 正在调用模型 |
| `generated` | 已生成但未保存 |
| `saved` | 已保存到 output |
| `failed` | 生成失败 |

## Output 行为

第一阶段只保存到 `output/`，不直接写入 Obsidian Vault。

保存时建议生成：

```text
output/YYYY-MM-DD-标题.md
```

如果模型给出的标题为空，使用分组 ID：

```text
output/2026-05-01-group-001.md
```

如果文件重名，自动追加编号：

```text
2026-05-01-标题-2.md
```

## 二次修改和再生成

用户可能会觉得第一次输出不好，因此需要支持二次迭代。

推荐流程：

1. 用户查看右栏 Markdown。
2. 如果不满意，在中栏说明中追加要求。
3. 点击重新生成。
4. 系统把原图、当前说明、上一版输出一起发给模型。
5. 模型返回新版 Markdown。

二次生成时的用户消息可以包含：

```text
这是上一版输出：
[上一版 Markdown]

请根据我的新要求改进：
[新的用户说明]
```

## MVP 功能范围

第一版只做必要功能。

| 功能 | 是否纳入 MVP |
| :--- | :--- |
| 本地网页打开 | 是 |
| 自动读取 `input_image/` | 是 |
| 图片缩略图预览 | 是 |
| 图片拖拽合并分组 | 是 |
| 每个分组独立说明 | 是 |
| 调用多模态模型生成 Markdown | 是 |
| 右栏编辑输出 | 是 |
| 保存到 `output/` | 是 |
| 状态持久化 | 是 |
| 直接写入 Obsidian Vault | 否 |
| 用户登录 | 否 |
| 云端同步 | 否 |
| 富文本编辑器 | 否，先用 textarea |
| 自动分类和打标签 | 由模型在 Markdown 中建议，不做独立功能 |

## 推荐实现技术

| 层 | 建议 |
| :--- | :--- |
| 前端 | React + Vite |
| 拖拽 | dnd-kit |
| 后端 | Node.js + Express，或 Python FastAPI |
| 模型 SDK | OpenAI SDK 兼容接口 |
| 状态存储 | `state.json` |
| 输出 | 写入本地 `output/` 文件夹 |

如果目标是最快做出可用原型，也可以先用 Node.js + Express + 原生 HTML/JS，等流程稳定后再升级 React。
